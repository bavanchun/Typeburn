## The Brutal Truth

This release felt *lean* after v1.0.0's friction. The plan audit (every claim verified against live code) eliminated scope creep early. But that cleanliness almost became complacency: two test fixtures (ui settings-cycle, storage normalize) were stale 2-theme artifacts that broke mid-implementation. The frustration: these tests *passed* in isolation but would have silently corrupted the test matrix if we'd shipped. Caught only because implementation forced re-exercise of the entire test surface. The lesson: test inheritance from prior phases is invisible debt until you touch the code. No amount of plan rigor catches that; only full integration does.

## Technical Details

**Theme pack implementation:**
- 6 theme packs as pure role→color maps (16 roles per theme, zero Tailwind class generation, zero config coupling).
- Approach A: core theme layering preserved (no config→theme import, maintains separation of concerns).
- Added luminance guard test: rejects light-palette-inversion edge case (e.g., inverted light palettes where lightRoles > darkRoles).

**DRY config↔theme sync test (Phase 2):**
- Single parametric test replacing 2+ instance-per-theme assertions.
- Verifies that every role in core theme appears in config's role list (forward direction).
- Maintains direction: config source-of-truth, theme implementation derives from it (no circular validation).

**Persistence-failure handling (Phase 3):**
- AppendHistory and SaveSettings errors were silently swallowed (lost to stderr, no trace).
- Non-blocking dismissible notice now surfaces these errors in View() without halting the session.
- View() refactored to single-return: all error paths converge to the same notice logic.
- Byte-identical on success path (no behavioral change if AppendHistory/SaveSettings succeed).

**Doc hygiene:**
- Removed stale README badge note referencing "v1.0.0 countdown" (post-1.0 baggage).
- Reworded dependency-layering rule in CONTRIBUTING.md to the *actual* invariant: "tests can import from tests, impl can import from impl, tests cannot import from impl" (was loosely phrased before).
- Fixed word-wrap comment (80-char → 100-char rule clarified).

**Code review (score 9.5/10, 0 critical):**

- No unused variables or logic dead-ends.

## Root Cause Analysis

Two stale test fixtures were a **Phase 2 blast radius**: 2-theme hard-coding from v1.0.0's limited theme set broke when 6 themes were added in Phase 4. Why not caught earlier?

1. **Test inheritance invisibility**: Tests written pre-expansion are implicitly coupled to the expansion scope. Static review of the test code doesn't expose the coupling until you *add* new themes.
2. **Fixture-level coupling**: Hard-coded theme names (e.g., `"light"`, `"dark"`) in test data assume a fixed theme universe. Only full integration (populating all 6 themes) triggered the mismatch.
3. **No cross-phase static check**: The plan's self-red-team caught most defects, but test coupling spans implementation phases — a gap in the gate.

## Lessons Learned

1. **Audit rigor + empirical exercise:** The plan audit (verify every claim against live code) eliminated most scope creep. But audit *alone* misses test coupling that only emerges during full-scope implementation. Integration is the final gate.

2. **Self-red-team of plans is cheap, high-value:** Pre-implementation red-team caught 4 real defects (strategy ambiguity, unused vars, parallel clarification, ownership). Cost: 30 min; prevented: merge churn, rework. Make it procedural.

3. **Fixture inheritance is invisible debt:** Test data written for a limited scope will silently break on expansion. Catch by: (a) naming test fixtures by intent, not example-theme (e.g., `generic-cycle-case` not `light-theme-cycle`); (b) full integration before merge.

4. **Disposable dry-run saved the release (again):** The v0.0.0-rc.test tag verified non-empty release notes (948 chars, not empty string) and correct v-prefixed asset naming before irreversible v1.1.0 tag. Same gate as v1.0.0 — it's a pattern, not a one-off.

5. **Protect-main hook gotchas (operational):**
   - `ck plan check` rewrites tracked plans/**/*.md mid-run (dirties worktree, blocks switch to main). Workaround: stash, housekeeping PR.
   - Hook evaluates HEAD before command, so chained `git switch -c && git commit` is denied. Fix: run switch as its own step.
