# Security Policy

## Reporting a Vulnerability

Please report security issues **privately** via GitHub's private vulnerability
reporting:

1. Go to the [Security tab](https://github.com/bavanchun/Typeburn/security)
   of this repository.
2. Click **"Report a vulnerability"** to open a private advisory.

Do **not** open a public issue for security-sensitive reports. You can expect an
initial acknowledgement within a reasonable time; please allow time for a fix
before any public disclosure.

## Supported Versions

Only the latest released `v1.x` line receives fixes. Security fixes are shipped
**fix-forward** as a new patch release (e.g. `v1.0.1`) — a published tag is
never deleted or re-pointed (the Go module checksum database is append-only).

## Binary Integrity — Unsigned Releases

Release binaries are **NOT cryptographically signed.** This is an intentional,
documented scope decision for v1, not an oversight.

Integrity is provided **only** by:

- **HTTPS transport** when downloading release assets from GitHub.
- **`checksums.txt`** (SHA-256), generated by the **same** release pipeline that
  builds the binaries and published as a release asset.

Implications you should understand before trusting a binary:

- `checksums.txt` lets you detect a corrupted or truncated download. It does
  **not** provide cryptographic proof of origin — it is produced by the same
  pipeline as the binaries, so it is not an independent signature.
- Trust therefore rests on the GitHub release pipeline and HTTPS, not on a
  detached signature you can verify offline.
- If you require signed artifacts, build from source at a pinned tag and inspect
  the `.goreleaser.yaml` / release workflow yourself.

To verify a download:

```sh
# from the directory containing the archive and checksums.txt
sha256sum -c checksums.txt --ignore-missing
```

Cryptographic signing (e.g. cosign/Sigstore) may be added in a future release;
it is explicitly out of scope for v1.0.0.
