## FINDINGS

### Finding 1 — Critical
Location: phase-01 §Requirements (releaseURLPrefix guard); phase-01 lines 26-30, plan.md:50.
Flaw: Plan claims the asset/checksum download is "guarded by the existing `releaseURLPrefix` constant." That constant is `https://github.com/bavanchun/Typeburn/` (cache.go:25) — a github.com host prefix. But the SAME phase mandates following a redirect to `objects.githubusercontent.com`. Once the redirect fires, the bytes actually arrive from a host the `releaseURLPrefix` prefix does NOT match. The guard validates only the *initial* URL the plan itself constructs (which is trivially official by construction), giving zero added safety, while the bytes-on-wire come from the CDN host the prefix can't express. The guard is theater.
Failure scenario: Reviewer/impl believes `releaseURLPrefix` constrains where bytes come from. It doesn't — it only re-checks a string the code just built. The real trust anchor is TLS + the `CheckRedirect` host allowlist, not `releaseURLPrefix`. Plan conflates two unrelated mechanisms.
Evidence: `internal/update/cache.go:25`; `internal/update/check.go:50-52` (prefix is used only to null out a URL already fetched). phase-01:26-30 asserts both guard-by-prefix AND CDN redirect.
Fix: Drop the `releaseURLPrefix` mention for downloads; state the actual control is (a) hardcoded official base URL construction + (b) `CheckRedirect` host allowlist. Don't claim a github.com prefix guards a githubusercontent.com download.

### Finding 2 — Critical
Location: phase-01 line 30 ("Allow exactly one redirect to `objects.githubusercontent.com`"); brainstorm-summary.md:72-73.
Flaw: The single-redirect-host assumption is asserted as fact with NO codebase or empirical evidence. GitHub release-asset downloads (`/releases/download/<tag>/<asset>`) do NOT reliably redirect to `objects.githubusercontent.com`. They have historically redirected through `github-releases.githubusercontent.com`, `objects.githubusercontent.com`, and AWS S3 (`*.s3.amazonaws.com` / `release-assets.githubusercontent.com`), and the target host has changed over time and varies by region/CDN. install.sh — the security bar the plan claims to match — does NOT hardcode any redirect host; it delegates to curl/wget default redirect following (install.sh:31 base URL + fetch helper).
Failure scenario: GitHub serves a 302 to `release-assets.githubusercontent.com` (current real behavior in many regions). The hardcoded `objects.githubusercontent.com`-only allowlist rejects it. `typeburn update` is then 100% broken in production for the exact happy path, on every platform, while all httptest-based unit tests (which use a fake server the test controls) pass green. This is the textbook "passes CI, breaks in prod" defect.
Evidence: install.sh:31,41-51 (no host allowlist on asset fetch — relies on curl/wget); no codebase fixture pins a redirect host; the plan cites none.
Fix: Either (a) allowlist the github.com + githubusercontent.com + amazonaws.com CDN families by suffix, not one exact host; or (b) match install.sh's actual posture — follow redirects but require HTTPS + final-response success + mandatory checksum (checksum is the real integrity gate, so the redirect host matters far less than the plan dramatizes). Add a non-hermetic integration test that hits a real release asset, or this assumption ships unverified.

### Finding 3 — High
Location: phase-03 §Requirements `classifyInstall`; brainstorm-summary.md:47-49.
Flaw: Managed-install classification is asserted to reliably distinguish brew/go/self-managed, but the heuristics are incomplete and the binary-name casing breaks the Go branch. (a) Windows package managers scoop (`~/scoop/shims`, `~/scoop/apps`) and chocolatey (`C:\ProgramData\chocolatey\bin`) are NOT handled at all — they'll be classified self-managed and the updater will try to swap a shim/shimmed exe. (b) The Go branch checks `<GOBIN>` / `<GOPATH>/bin`, but `go env GOBIN` is *empty by default* (only set if user exports it); the plan's selfpath_test only injects a non-empty gobin (phase-03:59), so the empty-GOBIN real-world case is untested and falls through to GOPATH/bin — fine only if GOPATH resolves, which the plan never resolves via `go env GOPATH` (it assumes an injected value). (c) The member extracted is lowercase `typeburn` (phase-02), but a `go install` binary is named `Typeburn` (README:58) — if classifyInstall ever false-negatives a go install (empty GOBIN + non-standard GOPATH), the updater would overwrite a `Typeburn` with a `typeburn`, and on case-insensitive macOS/Windows filesystems silently shadow/rename the user's binary.
Failure scenario: User `go install`ed with GOBIN unset and a custom GOPATH; classifyInstall returns SelfManaged; updater overwrites the go-managed binary, defeating the entire "refuse managed installs" design goal. Or a scoop user gets their shim clobbered.
Evidence: README.md:58-62 (capital Typeburn, GOPATH/bin); phase-03:56-59 (tests only inject populated gobin/gopath — empty-GOBIN and scoop/choco untested); no scoop/choco path in plan.
Fix: Resolve GOPATH via `go env GOPATH` fallback when GOBIN empty; add scoop/choco/winget path heuristics; add a default-to-refuse posture when classification is ambiguous (safer than false-self-managed); add empty-GOBIN test case.

### Finding 4 — High
Location: phase-04 step 2 of flow ("`update.Check(ctx, ver, force=true)`; nil result (dev) ... → already on latest"); phase-04:23-26.
Flaw: The plan treats `Check` returning `(nil, nil)` as equivalent to "already on latest, exit 0" in step 2, but step 1 already refuses dev/unknown/pseudo builds with a non-zero exit. `Check` returns `(nil, nil)` ONLY for dev/unknown/`v0.0.0-` versions (check.go:16-18) — the exact set step 1 just rejected. So the step-2 nil branch is dead code OR step 1's refusal is being silently bypassed. More dangerous: if step 1's refusal is loosened, a pseudo-version build hits `Check`→nil and the plan maps nil to "exit 0 already latest" (phase-04:23) — making `typeburn update` a SILENT NO-OP that exits 0 on a dev build, contradicting the success criterion "dev/unknown build refuses cleanly" (phase-04:83, success criteria line). The two requirements conflict within the same phase.
Failure scenario: A user on a pseudo-version build runs `typeburn update`, gets "already on latest (vX)" exit 0, and believes they're updated when the binary was never eligible. Silent failure masquerading as success.
Evidence: `internal/update/check.go:15-19` ((nil,nil) only for dev/unknown/v0.0.0-); phase-04:23 maps nil→exit0; phase-04:83 demands dev refuses. Contradiction.
Fix: Collapse to one rule: resolve version → if dev/unknown/pseudo, refuse (non-zero) BEFORE calling Check. After that point `Check` nil should be treated as an error/abort, never "already latest", because it's unreachable on a valid build and reaching it means a logic bug.

### Finding 5 — High
Location: phase-04 §Risk "detect non-tty stdin and refuse rather than hang" (line 94); phase-04:44-45 (prompt reads `cmd.InOrStdin()`).
Flaw: The non-tty detection is asserted but unspecified and untestable as designed. `cmd.InOrStdin()` returns an `io.Reader`, not an `*os.File`, so you CANNOT call `term.IsTerminal(fd)` / check `os.ModeCharDevice` on it — the test seam (`cmd.SetIn(bytes.Buffer)`) deliberately hides the fd. The plan wants both "tty-detect to refuse piped use" AND "inject a buffer for tests"; these are mutually exclusive through the same `InOrStdin()` reader. There is no stdlib reader-level "is this a tty" check.
Failure scenario: Implementer either (a) type-asserts `InOrStdin()` to `*os.File` — which breaks every unit test that injects a `bytes.Buffer` (panic/fallthrough), or (b) skips tty detection — and `typeburn update` in a CI pipeline / `echo | typeburn update` hangs forever waiting on a prompt, or reads EOF and proceeds with default-no but never errors clearly. Either way a stated requirement is unmet.
Evidence: cobra `cmd.InOrStdin()` returns `io.Reader`; phase-04:44-45 (read from InOrStdin) vs phase-04:94 (detect non-tty) — incompatible. No existing tty-detection helper in repo (grep: no `IsTerminal`/`ModeCharDevice` usage in internal/cli).
Fix: Decide explicitly: check `os.Stdin` fd directly for tty (separate from the injectable prompt reader), and require `--yes` when non-tty. Document that tty-detection uses real `os.Stdin`, while the prompt-read uses the injectable reader — two different inputs. Add an explicit test for the non-tty path.

### Finding 6 — Medium
Location: phase-05 §Risk "Binary-size cap ... net growth should be small" (lines 65-67); plan claims size-check is "the gate."
Flaw: "net growth should be small" is an unverified assumption presented as reassurance. Measured headroom is thin: current binary 8,681,490 bytes vs cap 10,485,760 (Makefile:16) = 1,804,270 bytes (17.2%) free. The new code links `archive/tar`, `compress/gzip`, `archive/zip`, `compress/flate`, `crypto/sha256`, plus cobra `update` wiring. `archive/zip`+`compress/flate`+`compress/gzip` are NOT currently linked (the existing binary only pulls `net/http`+`encoding/json` per Makefile:50 comment). zip/flate/gzip can add several hundred KB. The plan provides no measured estimate and explicitly says it won't "bump silently" — but it also hasn't measured, so it can't assert "small."
Failure scenario: Phase 5 `make size-check` fails at the very end of a 5-phase TDD effort; per plan's own rule that's "a real finding, not a number to bump" — meaning the whole feature is blocked at the finish line with no contingency. Should have been measured in Phase 2 (when archive deps land), not deferred to Phase 5.
Evidence: Makefile:16 (`SIZE_LIMIT ?= 10485760`), Makefile:50-54; measured `./bin/typeburn` = 8,681,490 bytes (17.2% headroom); phases 1-2 link tar/gzip/zip/flate/sha256 not currently present.
Fix: Move a size-check checkpoint into Phase 2 (right after archive deps land) so the cap risk surfaces early. State a measured delta budget. Pre-decide the contingency (raise cap deliberately vs. cut zip support) instead of discovering it at the gate.

### Finding 7 — Medium
Location: phase-03 §Requirements unix replace ("preserve original mode/owner where possible"); phase-03:30, 85.
Flaw: "preserve owner" via `os.Rename` of a fresh temp file is largely impossible without privilege and the plan hand-waves it. A new temp file created by the running user is owned by that user; you cannot `chown` it to a different original owner without root. For the common case where typeburn was installed by root into `/usr/local/bin` but the user runs `typeburn update` as themselves, the rename will fail with EACCES on the *directory* (can't write to `/usr/local/bin`) long before owner-preservation matters — and that's actually the brew-Intel `/usr/local` path, which classifyInstall is supposed to catch but only catches via `/Cellar/` substring (phase-03:24), NOT `/usr/local/bin` directly.
Failure scenario: Manual install to `/usr/local/bin` (owned root) by a user who used sudo once; classifyInstall sees `/usr/local/bin/typeburn` (no `/Cellar/`, no go path) → SelfManaged → updater attempts rename into root-owned dir → permission denied. Plan's success criterion "read-only/permission-denied target → clear error, original untouched" (phase-03:73) saves correctness, but the "preserve owner" promise is dead on arrival and misleads the implementer.
Evidence: phase-03:30 ("preserve original mode/owner where possible"), phase-03:24 (Homebrew detected by prefix OR `/Cellar/` — bare `/usr/local/bin` not classified managed); chown-other-owner requires root.
Fix: Drop "owner" from the promise (mode only). Explicitly document that root-owned install dirs surface a permission error telling the user to re-run with sudo or use their package manager. Note `/usr/local/bin` is a non-brew manual-install location that will hit this.
