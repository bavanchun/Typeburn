# Red-Team Security Review — typeburn update self-update plan

Perspective: Security Adversary + Fact Checker. All findings cite file:line.

## Finding 1 — Redirect SSRF guard models a control that does NOT exist; "inverse of FetchLatest" is false
- Severity: Critical
- Location: phase-01-download-and-verify.md (Architecture, lines 41-43; Risk "Redirect SSRF" lines 82); brainstorm-summary.md lines 18, 72-74
- Flaw: Plan asserts `FetchLatest` "blocks all redirects" and that Phase 1 is "the deliberate inverse." But `client.go:48-50` sets `CheckRedirect` to return `http.ErrUseLastResponse` — that does NOT block a redirect, it *stops following* it and hands the caller the raw 3xx response (Location header unfollowed). For a download that means a 302 yields an empty/HTML body, not a transfer. The plan's mental model ("block all redirects") is wrong, so the "allow exactly one redirect to objects.githubusercontent.com" design is being built on a misunderstanding of the existing primitive. A naive copy of the `client.go` pattern into download.go would silently download zero bytes on the real GitHub redirect; a "fix" that switches to following redirects without a strict host allowlist reopens SSRF.
- Failure scenario: Implementer copies `client.go`'s `CheckRedirect` verbatim (plan says "mirror the bounded-client pattern in client.go", phase-01 line 42). Download of the real asset (which 302s to objects.githubusercontent.com) returns the 302 body; sha256 fails or, worse, an implementer relaxes CheckRedirect to `nil` (follow-all) and an attacker who can MITM/poison the 302 Location redirects the fetch to an arbitrary host. The checksum still must match, but combined with Finding 4 (checksums fetched over the same channel) a redirect-controlling attacker swaps both.
- Evidence: `internal/update/client.go:46-50` (`CheckRedirect: func... return http.ErrUseLastResponse`); phase-01 line 42 ("Mirrors the bounded-client pattern in client.go"), line 43 ("inverse of FetchLatest's block all redirects"); brainstorm line 18 ("redirect-blocked").
- Suggested fix: Phase 1 must spell out the exact CheckRedirect: follow redirects but in CheckRedirect reject any `req.URL.Host` not in {`objects.githubusercontent.com`, `github.com`, `*.githubusercontent.com` exact-match list}, cap redirect count at 1-2, and require `https`. Add a test for the 302→allowed-host happy path actually transferring bytes (not just returning the 302). Correct the plan's false claim about `FetchLatest`.

## Finding 2 — checksums.txt is fetched over the SAME redirect-following channel; a redirect/MITM that swaps the archive can swap checksums.txt too
- Severity: Critical
- Location: phase-01 lines 16-18, 45-48 (download both archive + checksums.txt off the same base, same client); brainstorm lines 51-52, 79-82
- Flaw: The whole integrity story is "verify archive sha256 against checksums.txt." But checksums.txt is downloaded by the same bounded client, off the same `releases/download/<tag>/` base, subject to the same redirect policy. There is no independent trust anchor. If the redirect policy (Finding 1) is wrong/relaxed, or TLS is stripped, an attacker substitutes BOTH archive and checksums.txt consistently. The plan's own security caveat acknowledges this for *release compromise* but the plan presents sha256 verification as a download-integrity control while building a redirect path that can move the checksums source too.
- Failure scenario: Attacker controls the single allowed redirect hop or performs TLS MITM on a misconfigured proxy (`http.ProxyFromEnvironment` is inherited in `client.go:52` and will be copied). Both files are served from attacker host; sha256 matches attacker's archive; verification passes; malicious binary swapped in. Defense reduces to TLS only — exactly what the caveat admits, but the plan's success criteria ("sha256 mismatch aborts", phase-01 line 77) imply a stronger guarantee than exists.
- Evidence: `client.go:52` (`Proxy: http.ProxyFromEnvironment`); phase-01 lines 45-48 (both downloaded via same flow); brainstorm lines 79-82 (caveat).
- Suggested fix: Pin checksums.txt fetch to `github.com` host with NO redirect to the object CDN (release metadata files are served differently than asset blobs); reject `ProxyFromEnvironment` for the download client OR document that a hostile proxy fully defeats the control. State in phase-01 success criteria that the guarantee is "download-corruption integrity under trusted TLS," not tamper-proofing.

## Finding 3 — TOCTOU / privilege-escalation in atomic replace: temp created in target dir with no O_EXCL, no symlink check on target
- Severity: Critical
- Location: phase-03 lines 28-34 (replaceBinary: "temp in target's dir → chmod 0o755 → preserve original mode/owner → os.Rename"), Risk lines 81-85
- Flaw: Phase 3 describes creating a temp in the target's directory then `os.Rename` over the target, but specifies NO O_EXCL temp creation and NO symlink/regular-file check on the target before overwrite — unlike the codebase's own cache writer (`cache.go:112` uses `O_WRONLY|O_CREATE|O_EXCL, 0600` and comments "prevent symlink-race attacks") and unlike install.sh which explicitly rejects symlinks (`install.sh:143`). The install dir is `~/.local/bin` (`install.sh:33`) which on shared/multi-user or misconfigured systems can be world-writable or group-writable. An attacker who can write to that dir pre-creates the predictable temp name (or makes the target a symlink to a root-owned path) and wins the race.
- Failure scenario: (a) Predictable temp name + no O_EXCL: attacker pre-creates `<dir>/.typeburn.tmp` as a symlink to `/etc/cron.d/x`; replaceBinary opens it following the symlink and writes the new binary content there → arbitrary file write. (b) "preserve original mode/owner where possible" implies chown — if typeburn ever runs as root via sudo to update a root-owned binary, a symlinked target lets an unprivileged dir-writer redirect the chowned write. (c) `os.Rename` over a target that is itself a symlink replaces the link target, not the link.
- Evidence: `cache.go:108-114` (existing O_EXCL symlink-race defense the plan does NOT mirror); `install.sh:143` (`[ ! -L ... ] || die "refusing: ... is a symlink"`); `install.sh:145` (`stage="$BIN_DIR/.$BIN_NAME.$$.tmp"` uses PID — plan omits this); phase-03 lines 28-34 (no O_EXCL, no symlink check).
- Suggested fix: Mandate `O_CREATE|O_EXCL` with a PID/random suffix temp (mirror `cache.go:112` + `install.sh:145`); `lstat` the target and refuse if it is a symlink or non-regular file (mirror `install.sh:143`); drop the chown/"preserve owner" idea entirely (running as root + chown is the dangerous path) — only preserve mode bits. Add an explicit test for symlinked-target refusal and pre-existing-temp refusal.

## Finding 4 — Plan omits the symlink-in-archive guard that install.sh has; "path-traversal guard" is not the same control
- Severity: High
- Location: phase-02 lines 24-31 (path-traversal guard + single member); Risk lines 62-66
- Flaw: Phase 2 specifies a `..`/escape path-traversal guard and single-member extraction, but never says to reject the case where the extracted member ITSELF is a symlink or non-regular file. install.sh treats this as a distinct, necessary control (`install.sh:143-146`: refuse symlink, refuse non-regular-file). A tar entry named exactly `typeburn` can be a symlink (typeflag) or a hardlink; extracting it and then chmod 0o755 + swapping it in produces a binary that is actually a link. The plan's traversal guard does not cover member type.
- Failure scenario: A compromised/crafted archive (already in the threat model since binaries are unsigned) ships `typeburn` as a symlink to `/bin/sh` or a device file; extractBinary writes it as 0o755 (phase-02 line 27) and Phase 3 renames it into place; the "installed binary" is a symlink. On the tar path, `tar.Header.Typeflag` must be checked == `TypeReg`.
- Evidence: `install.sh:143` (`[ ! -L ... ]`), `install.sh:144` (`[ -f ... ] || die "...not a regular file"`); phase-02 lines 24-31 (only mentions `..`/escape, not typeflag).
- Suggested fix: In phase-02 require `hdr.Typeflag == tar.TypeReg` (and zip equivalent: reject entries whose mode has `os.ModeSymlink`); add a test with a symlink/irregular member. Bring Phase 2 to parity with `install.sh:143-144`.

## Finding 5 — Compare() permits downgrade/equal-version replace and silently equates malformed tags — `update --yes` becomes a downgrade primitive
- Severity: High
- Location: phase-04 lines 22-30 (Apply pipeline gated on `update.Check` UpgradeAvailable), brainstorm lines 49-52; relies on `Compare`
- Flaw: `compare.go:9-23` returns 0 (treated as "not newer", no upgrade) for malformed/equal, and `parseSemver` (`compare.go:30-46`) strips ALL suffixes so `v2.1.3-evil` compares equal to `v2.1.3`. Plan's gate is `Compare(currentVer, rel.TagName) < 0` (check.go:45). But the *download* step (phase-01) builds the asset URL from the tag the release advertises. If an attacker controls `releases/latest` (the disposable -rc dry-run tag risk install.sh explicitly defends at `install.sh:54-62, 100-103`), they can advertise a tag whose numeric core is higher but which points at an older/malicious archive. The plan's prerelease guard reuses `IsPrerelease`/`rel.Prerelease` (check.go:32) but does NOT replicate install.sh's *string-pattern* prerelease defense (`is_prerelease()` matching `-rc/-test/...`, `install.sh:54-62`) at download time.
- Failure scenario: Poisoned `latest` advertises `tag_name: v99.0.0` but `prerelease:false`; Check sees upgrade-available; Apply downloads `typeburn_99.0.0_..`. If that asset exists (attacker uploaded it to a compromised release) it installs. There is no "is this tag a stable, pattern-clean tag" string guard in the plan equivalent to `install.sh:54-62`.
- Evidence: `compare.go:13-15` (returns 0 on malformed), `compare.go:33` (`SplitN(v,"-",2)[0]` strips suffix); `check.go:31-33` (only struct-flag/IsPrerelease guard); `install.sh:54-62` + `install.sh:100-103` (string-pattern prerelease defense the plan omits).
- Suggested fix: Phase 4 must add the `is_prerelease`-style string guard on the resolved tag before download (reject `-rc/-test/-alpha/...`), and refuse when `Compare <= 0` is anything but strictly newer. Document that downgrade is not supported. Add a test: poisoned-tag with prerelease pattern → refuse.

## Finding 6 — Cache poisoning: forced check still WRITES cache, and Apply may read a poisoned cache to pick the download tag
- Severity: High
- Location: phase-04 lines 22-25 (`update.Check(ctx, ver, force=true)` then pipeline uses the Result); brainstorm lines 49-51
- Flaw: `Check(force=true)` bypasses cache on read (`check.go:23-27`) but still `cacheSave`s the result (`check.go:60`). The on-disk cache (`update-check.json` in StateDir, `cache.go:50`) is validated only by `validSemverRe` and `releaseURLPrefix` (`cache.go:75-82`) — it stores `Latest` (a tag string) and `ReleaseURL`. Nothing in the plan says Apply re-derives the download tag from a *freshly forced* fetch vs. trusting a cached `Result.Latest`. If `--check` (detect-only, phase-04 line 26) writes a cache and a later `update` (without examining whether force re-fetched) consumes `Result.Latest` to build the asset URL, a local attacker who writes a `validSemverRe`-passing `Latest` into the world-/user-writable state file steers the download tag.
- Failure scenario: Attacker with write access to the user's XDG state dir edits `update-check.json` `Latest` to `v9.9.9` (passes `validSemverRe`, `cache.go:74`). User runs `typeburn update`; if Apply trusts cached Latest, asset URL points at attacker-chosen tag. Even with force=true the SAVE path (check.go:60) means stale/attacker values persist for the version footer (Finding 7).
- Evidence: `check.go:60` (`_ = cacheSave(r)` even on forced path); `cache.go:74-76` (`validSemverRe` is the only Latest validation); `cache.go:50` (cache file in StateDir); phase-04 lines 22-25 (no statement that Apply ignores cache for tag derivation).
- Suggested fix: Phase 4 must explicitly state Apply derives the download tag ONLY from the live forced fetch Result returned in-process, never re-reads cache; and that the cache file is treated as untrusted (it already is for the footer). Add a test that a tampered cache file does not influence `update`'s chosen tag.

## Finding 7 — Asset/version normalization mismatch: `Latest` (tag) vs `assetName` (v-stripped) is unvalidated; injection guard not re-applied
- Severity: Medium
- Location: phase-01 lines 23-26 (`assetName` strips leading `v`); plan.md lines 52 ("GoReleaser strips the leading v"); phase-04 progress output "updated vX → vY" (line 31)
- Flaw: `Result.Latest` carries the raw tag (could be `v2.1.3`, or git-describe form, or—if cache-poisoned per Finding 6—an attacker string passing `validSemverRe`). Phase 1's `assetName` does `strings.TrimPrefix("v")` but the plan never says it RE-validates the tag against `validSemverFooter`/`validSemverRe` before interpolating it into the URL path (`releases/download/<tag>/...`, phase-01 line 28) or into human progress output. `validSemverFooter` (`screen_result_view.go:17`) and `validSemverRe` (`cache.go:24`) exist precisely because version strings reach rendering; the self-update path interpolates the tag into a URL and into stdout ("updated vX → vY") without a stated guard, potentially re-opening ANSI/control-char or path-segment injection that the footer guard closed.
- Failure scenario: A tag like `v2.1.3/../../other` or one with control chars reaches `assetURL` interpolation → request to an unexpected path; or printed to a terminal with escape sequences. The codebase already learned this lesson for the TUI footer (`screen_result_view.go:72`) but the plan does not carry the guard into the new output/URL path.
- Evidence: `screen_result_view.go:14-17,72` (validSemverFooter, applied before render); `cache.go:22-24` (validSemverRe); phase-01 lines 23-28 (assetName/assetURL build with no re-validation statement).
- Suggested fix: Phase 1 must validate the tag with the existing `validSemverRe` pattern BEFORE building any URL or printing it, and URL-escape the tag path segment. Add a test: malformed/control-char tag → refuse before any network call.

## Finding 8 — Windows rename-aside leaves a stale `.old` executable; downgrade/persistence + no integrity recheck
- Severity: Medium
- Location: phase-03 lines 31-32 (`os.Rename(target, target+".old")` → rename new in → best-effort remove `.old`), Risk lines 79-80
- Flaw: On Windows the running exe is renamed to `target+".old"` and the new binary moved in; `.old` cleanup is "best-effort on next run" (phase-03 line 32, line 80). This leaves a fully-functional old binary at a predictable path in a user-writable dir. There is no statement that the swapped-in binary's sha256 is re-verified AFTER the move (verification happens on the temp in Phase 1; between extract and the final rename there is a window). Also if the new rename fails after the aside-rename, the recovery path (restore `.old` → target) is not specified — risk of leaving the install with NO `typeburn.exe` and only `typeburn.exe.old`.
- Failure scenario: (a) `.old` persists; a user/script invoking the full path or an attacker re-points a shortcut runs the stale (possibly vulnerable) version indefinitely. (b) Power loss / lock between the two renames leaves `target` missing and only `.old` present; next-run cleanup logic deletes `.old` (best-effort) → binary gone. The plan has no rollback contract for the windows two-step.
- Evidence: phase-03 lines 31-32 (two-rename, best-effort .old), line 80 (cleanup deferred); contrast install.sh single atomic `mv -f` (`install.sh:147`) which has no aside-file residue.
- Suggested fix: Phase 3 must specify: re-verify the moved binary's sha256 post-rename (or at least size) before declaring success; define rollback (if second rename fails, rename `.old` back); place `.old` with a randomized name and attempt immediate delete, only deferring if the OS reports a sharing violation; never delete `.old` on next run until the primary is confirmed present and valid.

## Unresolved questions
- Does `update.Apply` consume `Result.Latest` (cache-influenced) or a freshly-returned live tag for asset-URL derivation? Plan is silent (Finding 6) — must be pinned before Phase 1/4 land.
- Will self-update ever run under sudo to update a root-owned target? If yes, the chown in phase-03 line 31 ("preserve original owner") is a privilege-escalation vector (Finding 3) and must be removed.
